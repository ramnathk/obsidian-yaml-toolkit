Examples ​

Real-world examples of YAML Manipulator rules.

Note: Examples show Condition → Action format. Enter these in separate fields in the plugin.

Content Management ​

Mark Drafts for Review ​

Input:

yaml
---
title: My Note
status: draft
tags: [project, active]
---

Condition: status = "draft" AND tags exists

Action (Rule 1): SET status "ready-for-review"

Action (Rule 2): SET reviewed_date "2025-11-24T21:45:00"

Output (after both rules):

yaml
---
title: My Note
status: ready-for-review
tags: [project, active]
reviewed_date: 2025-11-24T21:45:00
---

Multiple Fields

To set multiple top-level fields, create two separate rules with the same condition. Each rule performs one SET operation.

Archive Old Notes ​

Input:

yaml
---
title: Old Project
created_date: 2023-06-15
tags: [notes, project]
---

Condition: created_date < "2024-01-01" AND NOT tags contains "keep"

Action: APPEND tags "archived"

Output:

yaml
---
title: Old Project
created_date: 2023-06-15
tags: [notes, project, archived]
---

Multiple Fields

This example only appends a tag. To also add archived_date, create a second rule with action: SET archived_date "2025-11-24"

Add Tag to Untagged Notes ​

Output:

yaml

Tag Migration ​

Input:

yaml
---

---

Condition: tags contains "old-tag"

Action: REMOVE tags "old-tag", ADD tags "new-tag"

Output:

yaml
---
title: Project Note
tags: [project, active, new-tag]
---

Date Operations ​

Mark Overdue ​

Input:

yaml
---
title: Late Task
deadline: 2025-11-01
status: active
---

Condition: HAS deadline AND deadline < TODAY() AND status != "completed"

Action: SET status "overdue"

Output:

yaml
---
title: Late Task
deadline: 2025-11-01
status: overdue
---

Adding Tags Too

To also add the "urgent" tag, create a second rule with the same condition and action: ADD tags "urgent"

Conditional Workflow ​

Input:

yaml
---
title: High Priority Item
status: active
priority: 8
tags: [project]
---

Condition: (status = "active" OR status = "pending") AND priority >= 5 AND NOT tags has "reviewed"

Action (Rule 1): SET status "in-review"

Action (Rule 2): SET review_date "2025-11-24T22:10:00"

Action (Rule 3): APPEND tags "reviewed"

Output (after all three rules):

yaml
---
title: High Priority Item
status: in-review
priority: 8
tags: [project, reviewed]
review_date: 2025-11-24T22:10:00
---

Multiple Operations

This workflow requires three separate rules with the same condition, each performing one operation.

Working with Arrays of Objects ​

These examples show operations designed for arrays of structured objects (like tasks, meetings, or reading lists), not simple scalar arrays (like tags).

Key distinction:

  • Scalar arrays: tags: [project, active, urgent] - Use operations like ADD, REMOVE, SORT
  • Object arrays: tasks: [{name: "...", status: "..."}] - Use operations like UPDATE_WHERE, MOVE_WHERE, SORT_BY

These operations work inside arrays, modifying specific items based on their properties.

UPDATE_WHERE: Selective Updates Within Arrays ​

What makes UPDATE_WHERE unique: It updates fields within specific array items that match a condition. This is different from SET, which updates the note's top-level fields.

The Problem UPDATE_WHERE Solves:

When working with arrays of objects (tasks, meetings, inventory), you often need to update some objects based on their properties, not all of them. Regular operations can't do this:

  • SET tasks.status "done" - Only works if tasks is a single object, not an array
  • SET tasks[*].status "done" - Not supported (wildcard updates would change ALL items)
  • File-level conditions only filter which files to process, not which items within arrays

UPDATE_WHERE provides the solution: It adds a second level of filtering at the array item level.

Example: Task Management

Input:

yaml
---
title: Weekly Tasks
tasks:
  - name: Review PR #123
    status: pending
    priority: 8
    assignee: alice
  - name: Update documentation
    status: pending
    priority: 3
    assignee: bob
  - name: Fix critical bug
    status: done
    priority: 9
    assignee: alice
---

Rule: Flag high-priority pending tasks as urgent

Condition: (none - or optionally: tasks EXISTS)
Action: UPDATE_WHERE tasks WHERE status = "pending" AND priority >= 7 SET status "urgent", flagged true

Output:

yaml
---
title: Weekly Tasks
tasks:
  - name: Review PR #123
    status: urgent        # ← Changed (was pending, priority 8)
    priority: 8
    assignee: alice
    flagged: true         # ← Added
  - name: Update documentation
    status: pending       # ← Unchanged (priority too low)
    priority: 3
    assignee: bob
  - name: Fix critical bug
    status: done          # ← Unchanged (already done)
    priority: 9
    assignee: alice
---

Why you can't use SET: SET operates on the note level, not individual array items. You'd have to update ALL tasks or none.

More Examples:

Reassign all of Bob's tasks to Alice:

Action: UPDATE_WHERE tasks WHERE assignee = "bob" SET assignee "alice", reassignedDate TODAY()

Mark books as finished when fully read:

Action: UPDATE_WHERE readingList WHERE currentPage = pages SET status "finished", completedDate TODAY()

Cancel scheduled meetings with specific person:

Action: UPDATE_WHERE meetings WHERE status = "scheduled" AND attendees HAS "Bob" SET status "cancelled", cancelledDate TODAY()

SORT_BY: Organize Object Arrays ​

Example: Sort Tasks by Priority

Input:

yaml
---
title: Project Tasks
tasks:
  - name: Write tests
    priority: 3
  - name: Fix security issue
    priority: 9
  - name: Update README
    priority: 1
  - name: Review code
    priority: 7
---

Condition: (none needed)

Action: SORT_BY tasks BY priority DESC

Output:

yaml
---
title: Project Tasks
tasks:
  - name: Fix security issue      # ← priority 9 (highest first)
    priority: 9
  - name: Review code
    priority: 7
  - name: Write tests
    priority: 3
  - name: Update README            # ← priority 1 (lowest last)
    priority: 1
---

MOVE_WHERE: Prioritize Specific Items ​

What makes MOVE_WHERE unique: It moves items within an array based on their properties, allowing you to reorder without sorting.

The Problem MOVE_WHERE Solves:

When working with prioritized lists (tasks, watchlists, reading queues), you often need to move specific items to the top or bottom based on their properties, without fully reordering everything:

  • SORT_BY tasks BY priority - Reorders ALL items, losing manual arrangement
  • MOVE tasks FROM 2 TO 0 - Requires knowing exact indices (brittle)
  • Manual reordering - Can't be automated based on properties

MOVE_WHERE provides the solution: It selectively moves items matching a condition while preserving the order of non-matching items.

Example: Move High-Priority Items to Top

Input:

yaml
---
title: My Reading List
books:
  - title: Casual Read
    priority: normal
    pages: 200
  - title: Important Research
    priority: high
    pages: 450
  - title: Quick Reference
    priority: low
    pages: 50
  - title: Critical Paper
    priority: high
    pages: 30
---

Condition: (none needed)

Action: MOVE_WHERE books WHERE priority = "high" TO START

Output:

yaml
---
title: My Reading List
books:
  - title: Important Research    # ← Moved to top (priority: high)
    priority: high
    pages: 450
  - title: Critical Paper        # ← Moved to top (priority: high)
    priority: high
    pages: 30
  - title: Casual Read           # ← Stays below high-priority
    priority: normal
    pages: 200
  - title: Quick Reference       # ← Stays at bottom
    priority: low
    pages: 50
---

More Examples:

Move unwatched movies to the top:

Action: MOVE_WHERE watchlist WHERE watched = false TO START

Move completed tasks to the end:

Action: MOVE_WHERE tasks WHERE status = "done" TO END

Move urgent items after a specific item:

Action: MOVE_WHERE tasks WHERE priority = "urgent" AFTER name = "Planning meeting"

Difference from SORT_BY:

  • SORT_BY reorders the entire array by a field value
  • MOVE_WHERE moves only matching items, preserves order of others

Use MOVE_WHERE when: You want to prioritize/deprioritize specific items while keeping the rest in their current order.

Use SORT_BY when: You want a complete reordering by a specific field.

Combining Operations: Multi-Step Workflow ​

Example: Update and Organize Tasks

Input:

yaml
---
title: Sprint Tasks
tasks:
  - name: Bug fix
    status: pending
    points: 3
  - name: New feature
    status: in-progress
    points: 8
  - name: Code review
    status: pending
    points: 2
---

Step 1 - Mark small pending tasks as quick-wins:

Action: UPDATE_WHERE tasks WHERE status = "pending" AND points <= 3 SET status "quick-win"

Step 2 - Move quick-wins to top:

Action: MOVE_WHERE tasks WHERE status = "quick-win" TO START

Step 3 - Sort remaining by points:

Action: SORT_BY tasks BY points DESC

Final Output:

yaml
---
title: Sprint Tasks
tasks:
  - name: Bug fix             # ← Quick-win moved to top
    status: quick-win
    points: 3
  - name: Code review         # ← Quick-win moved to top
    status: quick-win
    points: 2
  - name: New feature         # ← Sorted by points
    status: in-progress
    points: 8
---

Understanding Array Types ​

Scalar arrays (simple values like strings, numbers):

yaml
tags: [project, active, urgent]
categories: [work, personal]
  • Operations: ADD, REMOVE, SORT, DEDUPLICATE, MOVE, etc.
  • Target: The values themselves

Object arrays (structured data with properties):

yaml
tasks:
  - name: Task 1
    status: pending
    priority: 8
  - name: Task 2
    status: done
    priority: 5
  • Operations: UPDATE_WHERE, SORT_BY, MOVE_WHERE
  • Target: Specific objects based on their properties
  • Also available: APPEND, PREPEND, INSERT_AT (for adding entire objects)

See Actions for complete operation reference.
